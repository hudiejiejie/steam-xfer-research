# TransferInventoryForm 控件命名与事件保留映射表

> 目标：在 UI-only 改造时，明确哪些控件名、事件绑定、数据绑定绝对不能断
> 原则：**保名优先、保事件绝对、保绑定第一**
> 说明：以下名称基于现有逆向材料与 WinForms 常规命名推导；真实名称以后续反编译结果为准

---

## 一、绝对优先保留的控件类别

如果你只打算改 UI，不打算碰核心逻辑，那么下面三类控件一定要优先保留：

### 1. 业务入口按钮
- 导入账号
- 导入 maFile
- 启动
- 停止
- 导出结果
- 查看库存

### 2. 参数输入控件
- 线程数
- 转移模式
- 接受方式
- 筛选方式
- 筛选值

### 3. 数据显示控件
- 主库号表格
- 待转号表格
- 日志控件

这些控件一旦改名、删掉重建、或丢失事件绑定，最容易让“界面看起来还在，但功能全断”。

## 二、推荐保留的控件名白名单

下面这份表，是当前最值得保住的名字。

### A. 主按钮区

| 作用 | 推荐保留控件名 | 绝对不要先做的事 |
|------|----------------|------------------|
| 导入账号 | `btnImportAccounts` | 改名为别的按钮名 |
| 导入 maFile | `btnImportMaFiles` | 删除重建再手绑 |
| 启动任务 | `btnStart` | 改成 `btnRunNow` 之类新名字 |
| 停止任务 | `btnStop` | 改事件名或换按钮对象 |
| 导出结果 | `btnExport` | 改成多个导出按钮而不保留原入口 |
| 查看库存 | `btnViewInventory` | 改成右键菜单后删原按钮 |

---

### B. 参数控件区

| 作用 | 推荐保留控件名 | 说明 |
|------|----------------|------|
| 线程数 | `numThreadCount` / `numThread` | NumericUpDown 常见 |
| 转移模式 | `cmbTransferType` | ComboBox |
| 接受方式 | `cmbAcceptMode` | ComboBox |
| 物品筛选方式 | `cmbItemType` | ComboBox |
| 物品筛选值 | `txtItemFilter` | TextBox |
| 代理/扩展输入 | `txtProxy` / `txtAppId` / `txtContextId` | 若存在则尽量保留 |

### C. 数据表格区

| 作用 | 推荐保留控件名 | 说明 |
|------|----------------|------|
| 主库号表格 | `dgvMaster` | DataGridView |
| 待转号表格 | `dgvBeTrade` | DataGridView |
| 日志控件 | `txtLog` | TextBox / RichTextBox / ListBox |

## 三、事件绑定保留映射表

这张表比控件名更重要。

### A. 主按钮事件

| 控件 | 事件 | 必须保留原因 |
|------|------|--------------|
| `btnImportAccounts` | `Click -> btnImportAccounts_Click` | 导入链入口 |
| `btnImportMaFiles` | `Click -> btnImportMaFiles_Click` | maFile 绑定入口 |
| `btnStart` | `Click -> btnStart_Click` | 总调度入口 |
| `btnStop` | `Click -> btnStop_Click` | 停止链入口 |
| `btnExport` | `Click -> btnExport_Click` | 导出入口 |
| `btnViewInventory` | `Click -> btnViewInventory_Click` | 库存查看入口 |

### 最关键提醒
如果你只是为了美化，**不要删掉旧按钮再新建同名外观按钮**。
因为 WinForms Designer 经常会：
- 新按钮长得一样
- 但事件没绑上
- 结果就是按钮能看不能点

最稳的做法是：
> **保留原按钮对象，只改外观、位置、尺寸、Text。**

---

### B. 参数控件事件

| 控件 | 事件 | 作用 |
|------|------|------|
| `numThreadCount` / `numThread` | `ValueChanged` | 更新线程数配置 |
| `cmbTransferType` | `SelectedIndexChanged` | 更新转移模式 |
| `cmbAcceptMode` | `SelectedIndexChanged` | 更新接受方式 |
| `cmbItemType` | `SelectedIndexChanged` | 更新筛选方式 |
| `txtItemFilter` | `TextChanged` / 读取时取值 | 更新筛选值 |

### 这里不要做的事
- 不要把这些控件替换成全新自定义控件后忘记转接原事件
- 不要只保留显示，不保留原读取逻辑

---

### C. 表格相关事件（高风险）

虽然我们还没拿到真实控件代码，但高概率会存在下面这些事件：

| 控件 | 事件 | 可能作用 |
|------|------|----------|
| `dgvMaster` | `SelectionChanged` | 更新当前选中主库号上下文 |
| `dgvBeTrade` | `SelectionChanged` | 更新当前选中待转号上下文 |
| `dgvMaster` / `dgvBeTrade` | `CellClick` / `CellDoubleClick` | 快速查看详情 / 操作 |
| `txtLog` | 无核心事件 | 主要是被动写入 |

### 建议
如果你没有完全确认这些事件的用途，先：
- 不删表格对象
- 不重建表格对象
- 只改列样式 / 外观 / 容器位置

## 四、数据绑定保留映射

### A. 主库号表格 `dgvMaster`

高概率绑定对象：`List<Account>` 或其包装 ViewModel。

#### 建议保留列语义
- 账号名
- SteamID
- 登录状态
- 待接受数
- 已分配数
- 上限
- maFile 状态

### 不要先做的事
- 直接改成新的自定义对象列表
- 只保留列标题，不核对绑定字段
- 用列索引替代绑定字段名后重排

### 因为最常见事故是
UI 改完后：
- 列看起来对了
- 但点击某行时代码还是按旧列索引读值
- 结果拿错账号 / 空值 / 直接报错

---

### B. 待转号表格 `dgvBeTrade`

高概率绑定对象：`List<Account>` 或任务态包装对象。

#### 建议保留列语义
- 账号
- 登录状态
- 库存数
- 可交易数
- 冷却数
- 已发报价数
- 当前状态
- maFile 状态

### 建议
如果要美化，优先：
- 改列标题文本
- 改列顺序
- 改列宽
- 改单元格颜色

而不是一开始改绑定字段名。

## 五、哪些控件可以“安全重建”

下面这些控件，通常可以安全新增或替换，因为它们不直接承担原业务入口：

| 控件类型 | 可否安全重建 | 备注 |
|----------|--------------|------|
| 顶部标题 Label | 可以 | 纯展示 |
| 状态摘要 Label | 可以 | 纯展示 |
| Panel 容器 | 可以 | 只影响布局 |
| 分组标题 Label | 可以 | 纯展示 |
| 详情区只读文本 | 可以 | 纯展示 |
| 图标/图片 | 可以 | 纯视觉 |
| 分隔线/装饰控件 | 可以 | 纯视觉 |

---

## 六、哪些控件不建议第一阶段重建

| 控件类型 | 不建议原因 |
|----------|------------|
| 启动/停止按钮 | 事件入口最关键 |
| 导入按钮 | 入口链容易断 |
| DataGridView 主体 | 绑定/选中/索引最容易出事故 |
| 日志主控件 | 原日志写法可能和类型绑定 |
| 参数 ComboBox / NumericUpDown | `Setting` 读取链容易断 |

这批控件最稳的做法是：
> **保留对象，只换皮。**

## 七、最推荐的改造手法

### 手法 1：保留对象，改属性
适合：
- 按钮
- 输入框
- 表格
- 日志框

你改：
- `Text`
- `Location`
- `Size`
- `Dock`
- `Anchor`
- `Font`
- `BackColor`
- `ForeColor`

### 手法 2：新增 Panel 包住旧控件
适合：
- 参数区
- 主库号区
- 待转号区
- 日志区

### 手法 3：新增展示控件，不接原逻辑
适合：
- 顶部状态条
- 详情摘要区
- 辅助信息区

---

## 八、第一阶段改造时的红线

### 红线 1
**不要一次性改控件名。**

### 红线 2
**不要删除旧按钮再新建同名按钮。**

### 红线 3
**不要先换掉 DataGridView 类型。**

### 红线 4
**不要在没确认事件前，把参数控件换成完全自定义控件。**

### 红线 5
**不要同时改布局 + 事件 + 数据绑定。**

否则一旦出问题，很难定位到底是哪层断了。

---

## 九、建议的实际操作顺序

### 第一步
保留所有原控件对象，先加 Panel 容器重排。

### 第二步
按这张表核对：
- 核心按钮名没变
- 核心按钮 Click 还在
- 参数控件读取事件还在
- 表格 Selection/Cell 事件还在（若有）

### 第三步
再改视觉层。

跑上一份《UI 第一阶段回归测试清单》。

## 十、这张表的实际用途

这份映射表不是为了“好看”，而是为了避免你在改 UI 时犯最常见的致命错误：

> **看起来只是换皮，结果把整条调用链悄悄剪断了。**

有了这张表，你后面就可以非常明确地知道：
- 哪些名字该守住
- 哪些事件绝对不能丢
- 哪些控件可以放心重做
- 哪些控件只能重排不能重建
