# UI 第一阶段回归测试清单（TransferInventoryForm UI-only 改造）

> 目标：在只改 UI、不动业务逻辑的前提下，确保 `TransferInventoryForm` 改造后核心功能不被改断
> 适用范围：`TransferInventoryForm.cs / Designer.cs / resx` 第一阶段改造
> 原则：**每改一块，立刻回归；每过一关，再继续动布局**

---

## 一、测试目标

这份清单不是测“界面好不好看”，而是测：

1. **事件有没有断**
2. **数据绑定有没有断**
3. **配置读写有没有断**
4. **日志回流有没有断**
5. **启动 / 停止 / 导入 / 导出 / 库存查看有没有断**

换句话说：

> **UI 改了，但引擎必须还是原来的引擎。**

---

## 二、测试前准备

### 必做准备
1. 备份以下文件：
   - `TransferInventoryForm.cs`
   - `TransferInventoryForm.Designer.cs`
   - `TransferInventoryForm.resx`
2. 确保原版程序在当前机器上能正常启动一次
3. 准备最小测试数据：
   - 1 个主库号账号
   - 1 个待转号账号
   - 对应 `.maFile`
   - 一份可导入的账号文本
4. 记录原版启动时的默认 UI 状态（截图最好）
5. 准备一个回滚点（git commit / 文件备份）

### 测试原则
- **不要边改边大面积重构**
- 一次只改一个区域：
  - 先参数区
  - 再表格区
  - 再日志区
  - 再按钮区
- 每改完一个区域，都跑完整小回归

## 四、最常见的 UI-only 改坏点

### 1. 控件改名
**后果**：事件函数、日志写入、读取参数全部失效

### 2. 控件删了重建
**后果**：Designer 新控件虽然长得一样，但原事件和绑定全丢了

### 3. 表格列顺序改了但代码还按索引取值
**后果**：点了 A 账号，后端拿到 B 账号或空值

### 4. 按钮文字改了，同时手贱改了按钮名
**后果**：按钮能看，点了没反应

### 5. 日志控件换类型
**后果**：原本 `AppendText` / `Items.Add` 的调用直接报错

### 6. Dock/Anchor 冲突
**后果**：控件逻辑在，但用户看不到，以为坏了

## 五、推荐测试节奏（非常重要）

### 最稳节奏
每做完一块，只测这 6 件事：
1. 能否启动窗体
2. 按钮还能不能点
3. 日志还会不会追加
4. 表格还显不显示数据
5. 启动还能不能触发
6. 停止还能不能触发

### 不要这样做
- 一口气把整个 Designer 改完再测
- 一次性把所有控件改名
- 先美化到很爽再验证逻辑

那样一旦坏了，很难定位到底哪一步弄断的。
